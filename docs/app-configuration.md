# App Configuration

`AppServiceProvider::boot()` is the application's configuration manifest. It calls six private methods in sequence — each owning a single concern, each independently reviewable. This pattern (sometimes called a Pinkary-style manifest) keeps `boot()` readable as a table of contents while preventing method bloat.

```php
public function boot(): void
{
    $this->configureModels();
    $this->configureCommands();
    $this->configureDates();
    $this->configureUrls();
    $this->configureVite();
    $this->configurePasswordValidation();
}
```

The sections below document each method in the order they are called.

## Models

```php
private function configureModels(): void
{
    Model::unguard();

    Model::shouldBeStrict(! $this->app->isProduction());

    Model::automaticallyEagerLoadRelationships();
}
```

**`Model::unguard()`** — runs in all environments. Disables mass-assignment protection globally, removing the need for `$fillable` or `$guarded` arrays on every model. Eloquent's mass-assignment guard predates modern form request validation; since this project validates all input at the request boundary, the model-level guard adds friction without adding safety.

**`Model::shouldBeStrict(!$this->app->isProduction())`** — enabled in all non-production environments (local, staging, testing). Strict mode throws exceptions for three classes of silent failure:

- **Lazy loading** — accessing a relation that was not eager-loaded raises a `LazyLoadingViolationException`, surfacing N+1 query patterns during development rather than silently degrading performance in production.
- **Silently discarding attributes** — assigning a column that does not exist on the model throws instead of quietly dropping the value, catching typos and schema mismatches early.
- **Accessing missing attributes** — reading an attribute that was not retrieved from the database (e.g., due to a `select()` limiting columns) throws instead of returning `null`, preventing subtle bugs from partial selects.

Strict mode is disabled in production because some of these violations are expected in rare edge cases that are acceptable at runtime but should never appear in development.

**`Model::automaticallyEagerLoadRelationships()`** — runs in all environments. Instructs Eloquent to detect when a relation is accessed on a collection without having been eager-loaded and automatically issue a follow-up query to load it. This eliminates N+1 query patterns globally without requiring developers to remember `->with()` on every query. The feature is enabled in all environments — including production — because preventing N+1 queries in production is more valuable than the conservative default of enabling it only locally.

## Commands

```php
private function configureCommands(): void
{
    DB::prohibitDestructiveCommands($this->app->isProduction());
}
```

**`DB::prohibitDestructiveCommands()`** — enabled in production only. Prevents Artisan commands that would irreversibly destroy database content — `migrate:fresh`, `migrate:rollback`, `db:wipe` — from executing when `APP_ENV=production`. Calling any of these commands in production raises a `ProhibitedFromRunningException` before any SQL is executed.

This is a safety net, not a deployment gating mechanism. The intended audience is a developer with production shell access who accidentally runs `artisan migrate:fresh` when they meant to run it against a local database. The prohibition costs nothing in normal operation and prevents a class of high-severity accidents.

## Dates

```php
private function configureDates(): void
{
    Date::use(CarbonImmutable::class);
}
```

**`Date::use(CarbonImmutable::class)`** — runs in all environments. Swaps Laravel's default `Carbon` implementation with `CarbonImmutable`. Every date operation (adding days, subtracting months, formatting) returns a new instance rather than mutating the original, making date math safe to pass across function boundaries without defensive copying.

The mutable default (`Carbon`) is a common source of subtle bugs: a method receives a date, modifies it in place, and the caller's date is silently changed. Immutable dates make this class of bug impossible — modifications always produce a new value.

## URLs

```php
private function configureUrls(): void
{
    if ($this->app->isProduction()) {
        URL::forceScheme('https');
    }
}
```

**`URL::forceScheme('https')`** — production only. Forces all URLs generated by Laravel's `url()` helper, route helpers, and signed URLs to use the `https` scheme, regardless of what scheme the incoming request used. This matters when the application sits behind a load balancer or reverse proxy that terminates TLS: the request arrives at the application container over plain HTTP, so Laravel would generate `http://` URLs without this call.

The gate is explicit (`if ($this->app->isProduction())`) rather than using the boolean argument form of `forceScheme` because URL scheme forcing only makes sense in production — local development runs over plain HTTP, and forcing HTTPS locally breaks the dev server.

## Vite

```php
private function configureVite(): void
{
    Vite::prefetch();
}
```

**`Vite::prefetch()`** — runs in all environments. Enables aggressive prefetching of frontend assets. When a page loads, Vite's prefetch strategy queues all other bundled chunks for background download, so that subsequent navigations feel instant — the JavaScript and CSS for the next page is already in the browser cache.

`Vite::prefetch()` with no arguments activates the most aggressive prefetch mode. The alternative API (`Vite::usePrefetchStrategy('aggressive')`) is not a public Laravel surface and should not be used directly — `prefetch()` calls it internally.

Prefetching is enabled in all environments rather than just production because the behaviour (and its absence) should be observable during local development. Asset sizes are small in development, so the overhead is negligible.

## Password Validation

```php
private function configurePasswordValidation(): void
{
    Password::defaults(fn () => Password::min(8)->mixedCase()->letters()->numbers()->uncompromised());
}
```

**`Password::defaults()`** — runs in all environments. Sets a global baseline password rule that any future validation using `Password::defaults()` will inherit:

- **`min(8)`** — minimum 8 characters.
- **`mixedCase()`** — at least one uppercase and one lowercase letter.
- **`letters()`** — at least one letter (redundant given `mixedCase()`, but explicit).
- **`numbers()`** — at least one digit.
- **`uncompromised()`** — rejects passwords that have appeared in known data breach lists via the [Have I Been Pwned](https://haveibeenpwned.com/Passwords) API (checked locally using k-Anonymity — the full password is never sent).

The global default exists so that future authentication forms calling `Password::defaults()` automatically inherit strong rules without each form needing to re-specify the policy. No auth scaffolding is implemented yet — this call establishes the contract that any future auth forms will follow.

Individual form requests can tighten (but not loosen) the rules by chaining additional constraints onto `Password::defaults()`.
